3.2 Comparing Standard vs Sub-directives

Ayisha.js provides two approaches to event handling: standard directives that execute expressions directly, and sub-directives that combine specific actions with trigger events. Understanding when to use each approach is crucial for building efficient applications.

<!-- Standard directives (direct expression execution) -->
<input @input="inputMsg = 'You typed: ' + event.target.value" placeholder="Type here for @input">
<div @hover="hoverMsg = 'Mouse is hovering!'" @leave="hoverMsg = 'Mouse left!'">Hover me for @hover</div>
<input @focus="focusMsg = 'Input focused with @focus!'" @blur="focusMsg = 'Input lost focus!'">

<!-- Sub-directives (action + event trigger) -->
<button @click="clickMsg = 'Set via @click at ' + new Date().toLocaleTimeString()">Click me</button>
<div @text:hover="'Text changed on hover!'" @text:leave="'Hover me'">Hover for text change</div>
<button @focus="focusMsg = 'Focus set via @focus!'">Focus me</button>

<!-- Complex example: Form with validation -->

🔍 Performance & Usage Comparison:

Standard Directives
  • @input: Complex validation logic
  • @hover: Dynamic calculations
  • @focus: Multi-step operations
  • Performance: Slower (full expression evaluation)
  • Flexibility: Maximum flexibility
  • Best for: Complex business logic
Sub-directives
  • @click: Simple value assignment
  • @text:hover: Direct text changes
  • @class:focus: CSS class toggling
  • Performance: Faster (optimized actions)
  • Flexibility: Limited but efficient
  • Best for: UI interactions & animations
💡 Pro Tip: Combine both approaches! Use sub-directives for simple UI changes and standard directives for business logic. This gives you the best performance and maintainability.

5.3 Loop Performance with @key

Why @key matters:

  • Without @key: Elements are recreated when array changes, losing state
  • With @key: Elements are tracked and reused efficiently
  • Best practice: Use unique, stable values like IDs for @key
❌ Bad: @key="$index" (index changes when items move)
✅ Good: @key="item.id" (stable unique identifier)

6.4. Chain Operations for Complex Data Processing

<!-- Multi-step data pipeline -->
<!-- Step 1: Filter products by category -->
<div @source="products" @filter="p => p.category === 'Electronics'" @result="electronics"></div>
<!-- Step 2: Add calculated fields -->
<div @source="electronics" @map="p => ({...p, totalValue: p.price * p.quantity, discount: p.price > 500 ? 0.1 : 0})" @result="processedElectronics"></div>
<!-- Step 3: Apply discounts -->
<div @source="processedElectronics" @map="p => ({...p, finalPrice: p.price * (1 - p.discount)})" @result="finalElectronics"></div>
<!-- Step 4: Calculate total savings -->
<div @source="finalElectronics" @reduce="(acc, p) => acc + (p.price * p.discount * p.quantity)" @initial="0" @result="totalSavings"></div>

10.3 Switch vs Multiple @if Comparison

When to Use Switch vs @if:

✅ Use @switch when:
  • Multiple mutually exclusive conditions
  • Checking the same variable against different values
  • Cleaner code structure for complex branching
  • Default fallback behavior needed
✅ Use @if when:
  • Simple boolean conditions
  • Checking different variables
  • Complex conditional expressions
  • Multiple conditions can be true simultaneously

15.2 Performance Monitoring and Tips

⚡ Performance Best Practices

❌ Avoid These Patterns:
  • Complex expressions in templates
  • Frequent @if/@show toggles
  • Missing @key in loops
  • Deeply nested conditionals
  • Large arrays without filtering
✅ Recommended Patterns:
  • Use @key for efficient list updates
  • Debounce frequent user inputs
  • Break complex logic into steps
  • Cache computed values in variables
  • Use @show for frequent toggles
🔍 Quick Debug Checklist:
  1. Check browser console for errors
  2. Use @state to inspect current data
  3. Verify @model bindings are working
  4. Test @key uniqueness in loops
  5. Validate @fetch API responses
